## Cíl

Umět **smazat uživatele** dvěma cestami:

1. **Sám uživatel** — v klientské sekci (nastavení profilu): „Smazat účet“
2. **Admin v God Mode** — u nahlášeného / porušujícího uživatele (`/mod/uzivatele` nebo z detailu profilu)

Obě cesty volají **stejný server-side pipeline** (anonymizace + odstranění auth účtu).

## UX požadavky

| Krok | Chování |
|------|---------|
| Potvrzení | Modal: akce je **nevratná**; u self-delete volitelně přepsat e-mail nebo checkbox „Rozumím“ |
| Po smazání | Odhlášení, redirect na `/` nebo `/login?message=account_deleted` |
| Admin | Povinný **důvod** (dropdown + volitelná poznámka) → `audit_events` (`profile_anonymized` / `profile_deleted_by_admin`) |

## Delete vs. anonymizace — rozhodnutí

**Nepoužívat** tvrdé CASCADE smazání celého uživatele z DB.

**Použít hybrid dle PRD §5.5** (bezpečné i pro GDPR):

| Entita | Akce |
|--------|------|
| `profiles` | Anonymizace PII (`email`, `name`, `surname` → hash / placeholder), avatar ze Storage |
| Inzeráty | Všechny aktivní/skryté → `deleted` (+ úklid fotek dle existujícího triggeru) |
| `comments` | **Zachovat.** Před smazáním auth: `author_nickname = '[smazaný účet]'`; FK `ON DELETE SET NULL` na `user_id` |
| `auth.users` | **Smazat** přes Supabase Auth Admin API — uživatel se může **znovu registrovat** stejným e-mailem |
| Audit / moderace | Záznamy (`audit_events`, `moderation_checks`, …) ponechat s anonymizovaným `actor_id` nebo NULL |

→ Re-registrace stejným e-mailem **není bezpečnostní riziko** — naopak to PRD explicitně předpokládá (zákaz změny e-mailu → „smaž účet a založ nový“).
